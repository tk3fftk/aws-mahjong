# 03. TDD ポリシー

本プロジェクトは **t_wada流の Red → Green → Refactor** で書いている。「テストが先、実装は後」というだけでなく、**仕様の問い直しサインとしてもテストの赤を使う**のがポイント。

---

## 基本サイクル

1. **Red**: 失敗するテストを書く
2. **Green**: **最小ステップ**で通す。最初は値をベタ書きしても構わない (`return ["1m","2m","3m"]` 等)
3. **Refactor**: 重複を取り除き、コードを整える

最小ステップにこだわる理由は、**「テストが本当に期待を検証しているか」をまず確認する**ため。いきなり一般実装を書くと、テストが甘い場合に気付けない。

## 三角測量

1ケースをベタ書き解で通したら、すぐ次のケースを追加して **「ベタ書きでは通らない」状態にする**。これを繰り返すうちに、自然と一般解へ昇格する。

例: `src/tiles.ts:mpszToTiles` の TDD 軌跡
- Red 1: `mpszToTiles("123m") === ["1m","2m","3m"]`
- Green 1: `return ["1m","2m","3m"]` (ベタ書き)
- Red 2: `mpszToTiles("7z") === ["7z"]` / 複数suit `"1m2p3s"` / 空文字 / ハイフン区切り を追加 → 当然 fail
- Green 2: 一般実装にリファクタ

## どの層をどの粒度でテストするか

| 層 | テスト方針 |
|---|---|
| 役判定 / 和了形分解 / 点数計算 / 鳴き適格性 (`yaku/`, `winning/`, `score`, `claims`) | **徹底的にテーブル駆動**。20-30ケースを目安 |
| 配牌・ツモなど薄い層 (`wall`) | 最小限。「136枚生成」「親14・子13×3、山残83」レベルで十分 |
| CPU (`cpu`) | 「ツモ和了可能なら win を返す」「不可ならランダム index を返す」+ 鳴き判断 (`decideClaim`) の基本パスのみ |
| `GameController` (`game`) | 統合テスト。「1局を seed 固定で完走させる」+ 仕込み壁での決定的シナリオ (後述のテストシーム参照) |
| UI (`ui/`) | **自動テスト無し**。`npm run dev` でブラウザ目視確認 |
| Scaffold / asset 取得 | TDD 対象外。`npm test` が「0件成功」で道具立てを検証する |

### GameController のテストシーム

ロン・ポン・チー・カンのような「特定の牌が特定の順で流れたとき」の振る舞いは、乱数まかせでは再現できない。`GameControllerOptions` の2つのシームで決定化する:

- **`wallFactory`**: 136枚の仕込み壁を注入する (省略時 `buildWall`)。全席の配牌とライブ壁先頭を mpsz 指定で組み立てるヘルパが `src/game.test.ts:riggedDeal` (配り順 index の対応はテスト内コメント参照)
- **`rng`**: CPU の打牌選択を差し替える。`() => 0` で「常に先頭 (ソート済み最小) の牌を打牌」になり、CPU の打牌が予測可能になる

この2つで「south が 8m を打牌 → 人間がロン」のようなシナリオを決定的に書ける。

## 仕様の問い直しサインとしての赤

実装の Bug でなく **仕様の仮定が間違っていたとき** も、テストの赤として現れる。これを「実装直して赤を消す」と捉えるのは危険で、**仕様を読み直し、テスト or 実装の意図を再定義する**機会だと考える。

### 実例1: 七対子と標準形の両立 (テストデータの修正)

書いたテスト:
```
it("七対子形 '11m 22m 33m 44m 55m 66m 77m' は標準形分解は失敗")
```

実行 → fail。

原因: `1122334455667 7m` は **234m + 234m + 567m + 567m + 11m** という標準形にも分解できる (順子2組×2、雀頭11m)。七対子と標準形が両立する珍しい手。

対応: テストの**仮定が間違っていた**。テストデータを `11m22m66m33p44s55z77z` に変更 (suit跨ぎ+字牌混じりで標準形不可にする) して七対子だけを成立させた。

### 実例2: isCombineAllowed の strict排他 (仕様解釈の修正)

最初は「false が出たら他のAWS役を除外」の strict排他で `detectAwsYakus()` を実装。

テスト:
```
it("マスター・レプリカ: 777s 刻子で 2飜")
it("静的サイトホスティング: 4p,5p,3s をすべて含むと 1飜")
```

実行 → fail。期待した役が `undefined` で返ってくる。

原因: `tile-superset` の判定はゆるく、1手で `web-application(1)` + `master-replica(2)` + `blue-green-deploy-kan(3)` のように複数役が同時にマッチする。strict排他で潰すと最高 han の1役のみ残り、テスト対象の役が消える。

対応: **実装直しではなく仕様解釈を見直した**。yaku.json description を読み直し「対応役との非複合」が本来の意味と判断、strict排他を撤回。詳細は [02-aws-yaku-judgment.md](./02-aws-yaku-judgment.md#iscombineallowedfalse-の解釈-重要)。

## ローカル運用

- `npm test` で 1度実行 (vitest run)
- `npm run test:watch` でファイル変更ごとに自動実行
- CI なし。`docs/app/` への build commit 前にローカルで `npm test && npm run build` を回す運用

テスト数の現状: 15ファイル / 221テスト (2026-06 時点、4人化+鳴き+ロン+ドラ実装後)。

---

## してはいけないこと

- **仕様変更で赤くなったテストを安易に消す**: そのテストが守っていた約束を理解する。残すべきなら新仕様に合わせて書き直す、不要なら commit log で「なぜ消したか」を残す
- **「テストを通すために実装をハック」する**: 例えば早期 return で specific ケースだけ通すパッチを当てる。三角測量で次のケースが必ず崩れる
- **UI を assert で固める**: 文字列の正確さや DOM 構造のテストは brittle すぎる。E2E は人間が触る (`npm run dev`)
